# Prime Atlas — session brief

US/UK property-investment SaaS. Next.js 14 App Router + Supabase (`vcnpevcmnobpznikahku`) + Stripe (live, acct SabioAI) + ScrapeOps. Deploy = push `main` (repo dahaninc/Prime-atlas → Vercel). Prod: https://prime-atlas-weld.vercel.app (domain not attached yet).

## Hard rules
- `npm run typecheck && npm run lint && npm test && npm run build` must pass before considering any change done. After deploy: `python3 scripts/qa/e2e_smoke.py` (39 checks).
- **Market scope FROZEN at 32 (18 US / 14 UK) until launch + revenue.** No new markets/feeds/geographies. Fix copy down, never DB up.
- **One pricing ladder**: free / explorer $29.99 / professional $69.99 / institutional $89.99, monthly or annual (−30%). Never sell a feature as a standalone add-on SKU. `src/lib/entitlements.ts` is the single source of truth for every tier-gated cap/feature — every gated route/action reads from it, never an ad hoc `tier !== "free"` check.
- **Never advice language** ("good deal", rankings-as-recommendation). Deltas/calculations + "not investment advice". Say "institutional-grade intelligence", NEVER "insider information".
- Server-side gating for anything paid; UI locks are presentation only. Redaction/blur must strip data server-side (blurred DOM text = leak).
- No caching service worker (sw.js is a self-destruct kill switch, no-store). Never re-enable `typescript.ignoreBuildErrors`.
- Dark theme tokens only (`bg-card`, `text-foreground`, `text-zinc-400/500`); Prime Blue `bg-primary` = actions only; `pa-green` = live/positive data only. `toast()` not alert().
- **Never run `vercel env pull`** (or any Vercel CLI mutating command) in a disposable/background bash one-liner — silently overwrites `.env.local`; wiped secrets (except public Supabase values) are unrecoverable via CLI/dashboard.
- **Never hand-roll price/currency formatting or reimplement an engine.** Import `fmt`/`symFor` from `src/lib/money.ts`; import real engines (`comps.ts`, `realYield.ts`, `screener.ts`, `marketReport.ts`) into any one-off script that renders a "here's what the product shows" artifact. Never reimplement their math.
- **Every money column is MINOR UNITS (cents/pence).** Any SQL/TS bound against a price literal must be in minor units. Before "fixing" a price constant, divide by 100 and sanity-check against a real listing — a 2026-07-08 near-miss (see `supabase/migrations/016`/`017`) briefly "fixed" a correct ceiling by misreading cents as dollars; caught before it hit prod state, reverted same session.
- **No number ships without a real basis.** If a metric can't be computed from real data, show "insufficient data" — never a heuristic, hardcoded formula, or placeholder value that looks like a real one. This is the product's core differentiator and the subject of most of this session's work (see Methodology below).

## Methodology — the actual product
- **Discount-to-value = ZIP-level comps, never a blended metro median.** `src/lib/comps.ts` (`screenByZipComps`) computes each listing's discount vs the median $/sqm of ≥5 comps in the same ZIP × property_type × bedrooms. Below 5 comps → "insufficient comparable data", never a fallback. Beyond ±60% = presumed data artifact, excluded from ranking. Every ranked deal carries its comp evidence (addresses/prices/$/sqm), auditable not asserted. **This is the ONE discount engine in the codebase** — `src/lib/server/compScreens.ts`'s `fetchZipCompScreens` is a thin wrapper around it (paginated fetch + same sanity bounds), never a reimplementation. Verified 2026-07-09: Market Feed detail, Deal Board (This Market + All Markets) all call one or the other — cannot disagree about the same listing.
- **Coverage reality (2026-07-09, US only — UK is 0% everywhere, no usable size_sqm/postcode)**: 8 of 18 US markets have any ZIP-comp coverage at all — Charlotte (46/161 listings), Indianapolis, Columbus, Denver, Phoenix, Houston, Boston, Nashville. Flagship metros (SF/LA/Chicago/Austin/Seattle/Miami) are at 0. Widening per-market listing density is the TOP data-roadmap priority (KNOWN_ISSUES.md) — the methodology is only as good as comp density behind it.
- **Real yield**: `src/lib/realYield.ts` (`computeRealGrossYieldPct`) gates on ≥10 real rent comps (`market_rent_stats`, mig 014) — never a fallback/national-average estimate. Replaced the deleted `src/lib/yield.ts` (a hardcoded US-state/UK-city rent lookup table) 2026-07-09 — that heuristic had been copy-pasted into 4 separate places (market-feed detail, deal-alerts cron, contact-request's research-report email, and the email template itself) before being found and closed in one pass. **Yield coverage and discount coverage are almost disjoint sets** — SF/LA/Seattle have real yield but zero discount coverage; Charlotte/Indianapolis/etc. have discount coverage but zero yield. No single US listing today has both real.
- **Financing scenarios**: `computeScreener` (`src/lib/screener.ts`) — same engine for Deal Board's Preliminary Underwrite panel, market-feed detail's Financing Scenarios panel, and the Deal Brochure. DSCR/cash-on-cash/exit-value require a real rent basis (read "n/a" without one); Monthly P&I is always real (price-only, no rent dependency).
- **`buildMarketReport`** (`src/lib/marketReport.ts`): real demand-signal narrative from municipality scores + live stats. `countryMedianPpsqm` is optional — only `/reports/market`'s `actions.ts` computes it for real (a genuine cross-market aggregate query); every other caller omits it rather than passing null. No momentum/trend signal fires without ≥2 real `market_score_history` snapshots (system has exactly 2 snapshot days total as of 2026-07-09 — thin, by design suppressed in 6 of 7 surfaces that use this engine).
- **KNOWN GAP, not yet fixed**: `/reports/market`'s `actions.ts` still computes its "underpriced"/mispricing count from the BLENDED metro median (`market_listing_stats.underpriced_count`), not the ZIP-comp engine above. Confirmed live 2026-07-09: for Charlotte, this surface would show 45 vs. Deal Board's real 9 — a 5× in-app contradiction for any logged-in user. Migration is scoped (swap in `fetchZipCompScreens` for that one municipality, ~30 lines in `actions.ts`, no new engine) but NOT built. Its public-sharing exposure IS closed (see next line) — the in-app surface is not.
- **Report/screener sharing** (`/s/[token]`, `src/app/actions/share.ts`): screener-analysis sharing (`kind: "analysis"`) is real and unaffected (own engine, no blended-median dependency). **Market-report sharing (`kind: "report"`) is DISABLED for launch (2026-07-09)** — `createShareLink` rejects it server-side before touching the DB; `ReportClient.tsx` has no share button; existing/future `/s/[token]` report links render a clean "temporarily unavailable" card, never the live (possibly contradicting) payload. Re-enable once the gap above is migrated.
- **Data-integrity stopgaps** (mig 014/015/017 + `src/lib/listingSanity.ts`): `onthemarket` corrupts ~99% of scraped rent prices; `zillow` corrupts a smaller share of sale price/size_sqm/bedrooms (confirmed: silently moved LA's median 43% before the fix). Migrations bound the aggregate views; listingSanity.ts bounds raw per-listing display. **Scraper itself still writes bad rows every run** — root-cause fix is an open follow-up. Also: 2 confirmed municipality-geocoding mismatches (street name coincidentally matches a city name).
