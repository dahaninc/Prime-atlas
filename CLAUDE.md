# Prime Atlas — session brief

US/UK property-investment SaaS. Next.js 14 App Router + Supabase (`vcnpevcmnobpznikahku`) + Stripe (live, acct SabioAI) + ScrapeOps. Deploy = push `main` (repo dahaninc/Prime-atlas → Vercel). Prod: https://prime-atlas-weld.vercel.app (domain not attached yet).

## Hard rules
- `npm run typecheck && npm run lint && npm test && npm run build` must pass. After deploy: `python3 scripts/qa/e2e_smoke.py` (39 checks).
- **Market scope FROZEN at 32 (18 US / 14 UK) until launch + revenue.** No new markets/feeds/geographies. Fix copy down, never DB up.
- **One pricing ladder**: free / explorer $29.99 / professional $69.99 / institutional $89.99, monthly or annual (−30%, see below). Never sell a feature as a standalone add-on SKU — but the ladder itself now has real per-tier quotas/capabilities (screener runs, OM parsing, contact reveals, bulk export), not "unlimited on any paid tier." **`src/lib/entitlements.ts` is the single source of truth for every tier-gated cap/feature — every gated route/action reads from it, never an ad hoc `tier !== "free"` check.**
- Never advice language ("good deal", rankings-as-recommendation). Deltas/calculations + "not investment advice". Say "institutional-grade intelligence", NEVER "insider information".
- Server-side gating for anything paid; UI locks are presentation only. Redaction/blur must strip data server-side (blurred DOM text = leak).
- No caching service worker (sw.js is a self-destruct kill switch, no-store). Never re-enable `typescript.ignoreBuildErrors`.
- Dark theme tokens only (`bg-card`, `text-foreground`, `text-zinc-400/500`); Prime Blue `bg-primary` = actions only; `pa-green` = live/positive data only. `toast()` not alert().
- **Never run `vercel env pull` (or any Vercel CLI mutating command) inside a disposable/background bash one-liner.** It silently overwrites `.env.local`. Once wiped every real secret except public Supabase values is unrecoverable via CLI/dashboard (Vercel "Sensitive" vars are write-only) — only the owner's own password manager / each service's dashboard can restore them.
- **Never hand-roll price/currency formatting.** Import `fmt`/`symFor` from `src/lib/money.ts`. A throwaway review script once reimplemented this with a wrong divisor and showed a real $137,500 listing as $1.38M — the bug never touched production, but only because production already used the shared function. Any future one-off script that renders a "here's what the product shows" artifact must import real formatters/engines, never reimplement them.
- **`properties.price` (and every money column) is MINOR UNITS — cents/pence.** Any SQL/TS bound, ceiling, or comparison against a price literal must be written in minor units (mig 015's `5000000000` = $50M, correct; listingSanity's `5_000_000_000` = $50M). Two real incidents on 2026-07-08: a "$5B typo fix" briefly applied to prod that was actually a 100x-tightening misread (reverted, mig 016/017 tell the story), and a genuine one-zero-short `MAX_SANE_PRICE` that silently nulled 116 real $5M–$50M listings. Before "fixing" any price constant, divide by 100 and sanity-check against a real listing.

## Architecture (1 line each)
- Migrations `supabase/migrations/` 001–015 = live (014/015 are data-integrity stopgaps, see below). After schema change: regenerate `database.types.ts` (MCP generate_typescript_types).
- Auth: FIRST-PARTY progressive-enhancement forms — `<form action={serverAction}>` in AuthForm → src/app/auth/actions.ts (zero-JS safe — real no-JS form POST works for manual/scripted testing too, see Dev environment below).
- Middleware protects /dashboard /watchlists /signals /opportunities/finder /admin /portfolio /deal-board /screener /reports/market; paid-only: /signals /opportunities/finder. Admin gate: src/lib/auth/admins.ts `isAdminEmail()` is the ONLY source of truth (ADMIN_EMAILS env override). Admin-panel access is orthogonal to subscription_tier — an admin account is NOT automatically Institutional; set tier separately via /admin's TierSelect.
- profiles column grants (mig 004): client can update only full_name/alert_preferences — payment_method_on_file, tiers etc. are service-role-only. Admin tier override UI: /admin (TierSelect + setUserTier action), no more manual SQL.
- Stripe: webhook resolves user via subscription metadata user_id; 500 on DB failure → retry; all billing states handled. Card-on-file activation: /api/stripe/setup gates free screener quota (mig 008, unchanged — free tier only, capped at 3/mo). Annual billing (−30%, 3 extra live prices, lookup_key `prime_atlas_<tier>_annual`, no new env vars) — webhook tier resolution falls back to price.metadata.tier when a price ID isn't in the env-var map.
- Public pages static/ISR via cookie-less `public.ts`; never add cookies() to them. Static pages redact listings unconditionally.
- Free-tier funnel: non-members get locality-only addresses + zero photos (src/lib/access.ts, fails closed); share links /s/[token] (mig 011, unguessable tokens). /underpriced tiered via entitlements: free = aggregate stats + waitlist only, explorer = real top-3 teaser (ranked, not fabricated), professional+ = full feed.
- Screener: OM/rent-roll/T12 parsing is Professional+ only (entitlements-gated; free/explorer see an upgrade panel, not the dropzone). Screener runs capped per tier (free 3/mo unchanged, explorer 5/mo, professional 100/mo, institutional unlimited). Parsed fields carry per-field confidence + source citation (Claude-returned, never fabricated) — shown as color-coded dots in the UI. Screener analysis export (`/api/export/screener`, Professional+) is a separate deliverable from the Deal Board export below.
- Contact reveals are quota-metered per tier via `checkContactRevealQuota` (free 0, explorer 10/mo, professional 100/mo, institutional unlimited) — previously binary member/non-member with no cap.
- Deal Board "Investment Analysis Report" export (`/api/export/ic-memo`) is Institutional-only (was: any paid tier). Rendering lives in `src/lib/icMemoTemplate.ts` (pure, vitest-tested, extracted from the route). It is a MARKET-SCREENING memo — deliberately has NO ground-up development pro-forma (that stays on-screen only in the "Preliminary Underwrite" panel, explicitly labeled as not exported; the two are different investment strategies and mixing them read as incoherent). Gross yield only renders with ≥10 real rent comps for that market (`market_rent_stats`, mig 014) — otherwise "insufficient rent data", never a fallback/national-average estimate.
- **Discount-to-value = ZIP-level comps, never a blended metro median** (2026-07-08 methodology rebuild, owner-approved): `src/lib/comps.ts` computes each listing's discount vs the median $/sqm of ≥5 comps in the same ZIP × property_type × bedrooms (`MIN_ZIP_COMPS=5`, band 15–60%, beyond ±60% = presumed artifact). Below 5 comps → "insufficient comparable data", NEVER a metro-median fallback — rigor over coverage is the owner's explicit call. Every ranked deal in the Deal Board + memo carries its comp evidence (addresses/prices/$SF) so the discount is auditable. UK = structurally insufficient (0% size/postcode) until the scraper backfills; UK keeps real rent-yield where covered. The memo's Section-1 mispricing count uses this same screen (`zipMispricingCount` from /api/deal-board/listings — the route also snapshots market_listing_stats in the same response so all memo sections read one query instant). Coverage reality: flagship metros (SF/LA/Chicago/Austin/Seattle/Miami) are at 0 covered listings today — widening per-market listing density is the TOP data-roadmap priority (KNOWN_ISSUES.md). ALL user-facing "underpriced" surfaces run on this basis via `src/lib/server/compScreens.ts` (shared paginated-fetch helper): /underpriced (covered-US-submarkets only, honest coverage line), deal-alert emails, Deal Board row chips. Pricing-page copy qualifies the feed as "covered US submarkets" — never claim flagship/UK discount coverage. Only deliberate exception: the on-screen Deal Board macro-read dispersion signal, labeled "below median /sqm".
- Engines (pure + vitest): screener.ts, marketReport.ts, levers.ts, proforma.ts (ground-up dev math — NOT part of the Investment Analysis Report export, on-screen Deal Board only), yield.ts, icMemoTemplate.ts, entitlements.ts, money.ts, listingSanity.ts, comps.ts.
- **Data-integrity stopgaps (mig 014/015 + src/lib/listingSanity.ts)**: `onthemarket` provider corrupts ~99% of scraped rent prices (concatenation-style parsing bug); `zillow` provider corrupts a smaller share of sale prices/size_sqm/bedrooms (confirmed: this silently moved LA's published median price 43% before the fix). Migrations exclude/bound these in the aggregate views; listingSanity.ts bounds them on raw per-listing display (market-feed, market-feed/[id], deal-board). **None of this fixes the scraper itself — it still writes bad rows every run.** Root-cause fix is an open follow-up (see State/next). Also found: 2 confirmed cases of a property assigned to the wrong municipality when a street name coincidentally matches a city name (e.g. "N Raleigh Street" in Denver → assigned to Raleigh, NC) — separate bug, scraper's geocoding step, not price parsing.
- Market feed (/market-feed): fetches ALL active listings per (currency, listing_type) segment via paginated `.range()` — PostgREST hard-caps any single request at 1000 rows regardless of `.limit()`. Don't reintroduce a flat `.limit(N)` global query.
- Scraping: /api/cron/scrape-listings tiered by market size — `?tier=daily` (busiest: top-10 US sale + top-5 US/UK rent metros + London, existing cadence) vs `?tier=longtail` (rest, every 3 days, vercel.json `*/3` day-of-month). Never state this cadence split in user-facing copy. enrich-agents also every 3 days (was hourly — cost driver, not freshness-critical). ~11k properties.
- Opportunities: all 32 markets ≥3 active theses; /api/cron/generate-opportunities is grounded-Claude, dedupe-indexed (mig 010), self-terminating. `opportunities.source_name`/`source_url` are often self-sourced ("Prime Atlas Intelligence", url null) — anywhere a thesis is surfaced to a user/export, show the source line so a self-sourced claim reads as internal analysis, not an external citation.
- Anthropic responses may lead with a thinking block — always `content.find(b => b.type==="text")`. Cron routes need `fetchCache = "force-no-store"`.

## Dev environment
- **Claude_Preview and claude-in-chrome are both unreliable in this sandbox** (preview: `EPERM`/`uv_cwd` on server spawn; claude-in-chrome: extension frequently not connected). Not a codebase issue — don't spend time "fixing" it in-repo.
- Workaround for manual/scripted verification of a real running instance: start the dev server via plain `npm run dev` in Bash (check `lsof -i :3000` first — orphaned dev servers from earlier in a long session are a real recurring issue, safe to `kill` if `ps -p <pid> -o lstart,command` confirms it's your own stray `next-server`). Auth: the login form is genuinely zero-JS-safe — GET `/auth/login`, scrape the `$ACTION_REF_1`/`$ACTION_1:0`/`$ACTION_1:1`/`$ACTION_KEY`/`redirectTo` hidden field values, POST them back with `email`/`password` as a real multipart form; the response sets a real Supabase session cookie usable against any authenticated route via curl.
